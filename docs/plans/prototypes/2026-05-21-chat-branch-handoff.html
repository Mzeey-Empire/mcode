A single-page walkthrough of every scenario discussed during the grill. Renamed from "chat branch" to "chat fork" to disambiguate from git branches (which forks may also create). The fork flow reuses the existing parent composer in fork mode — no separate child composer is introduced. Each row reads left-to-right as a temporal flow.

Scenario 10 — Edge cases discoveries

Edge cases the prototype surfaces

Scenarios the visual exercise surfaced that the prose plan did not pin down. Each one needs a decision before implementation.

edge · 01
Parent is mid-stream when user clicks branch

Parent's sdkSessionId is in use by an active turn. Resuming it for path B may race with the parent's own stream.

// proposal: wait for parent stream to settle before kicking B, with a 30s timeout fallback to D
edge · 02
Parent has no sdkSessionId yet

User branches from their very first message before the assistant has replied. No session to resume.

// proposal: skip B/A entirely, go direct to D with reason=null (not an error, just unavailable)
edge · 03
Side-channel query times out

Provider accepted the request but the stream stalls beyond 60s. User watching the skeleton gets impatient.

// proposal: 60s soft-cap; abort + fall to D; classifier returns "transient"
edge · 04
User abandons the child while generating

Click branch, then navigate away to a different thread, close the app, etc. Pipeline should still finish or be cleanly cancelled.

// proposal: pipeline runs server-side independent of UI; if child is deleted before completion, drop result
edge · 05
Fork message has attachments larger than disk budget

Parent message included a 50MB screenshot. Straight-copy duplicates it; multiple forks multiply.

// proposal: cap individual attachment copy at 25MB; skip + log if larger; reference parent path instead
edge · 06
Two users branch from the same parent simultaneously

Not applicable for single-user desktop today, but path A's hidden turns are not idempotent — concurrent invocations would interleave on the same session.

// proposal: per-thread mutex on path A; B is already safe via resume's fork semantics
edge · 07
Path A's disregard turn doesn't take

Cursor follows instructions usually but not always. The parent's next real reply references "the handoff I just wrote."

// accepted: documented limitation. Adding more disregard turns hurts cost without reliable benefit.
edge · 08
Child uses a different provider than parent

User branches from a Claude thread but picks Codex for the child. Budget is sized to Codex's cap; generation runs on the parent (Claude).

// already handled: budget pulled from childProvider.maxInputCharactersPerTurn
edge · 09
Handoff doc itself exceeds child cap

Provider ignored the budget and produced 5k chars when we asked for 2k. Inline injection will fail on the child's first turn.

// proposal: post-write validation; if over cap, truncate at section boundaries + emit warning chip
edge · 10
User keeps typing during gated send window

Typed message gets queued, handoff lands, queued message fires automatically. Surprising if the user thought their typing was being ignored.

// proposal: subtle "queued · sends on handoff ready" hint replaces send-blocked text
edge · 11
Slash command typed before provider switch fully resolves

User changes provider, slash palette is mid-refresh, user hits ⏎ on a command that exists in the old set but not the new one. Send fires under the new provider with a non-existent command.

// proposal: validate command against effective-provider's set at send-time; if missing, // toast "/cmd not available on codex" and // hold the message in the textarea
edge · 12
Same-command, different behavior across providers

Both Claude and Codex expose /verify, but the underlying scripts differ (Claude reads .claude/settings.json, Codex reads .codex/hooks.json). User reasonably expects the same behavior after a provider switch.

// accepted: this is a doc problem, not a UI problem. Each provider's /verify is its own // script with its own pre/post hooks. The slash palette's caption already hints // (".codex/prompts/" vs Claude default).
edge · 13
ComposerForkBar persists across thread switches

User clicks fork, then navigates to another thread before submitting. Does the fork bar stay on the parent, follow the user, or get cleared?

// proposal: per-thread fork-mode state. Bar stays on the originating thread. Navigating // back restores. Navigating away does NOT carry it to other threads' composers.
Scenario 11 — Open questions decisions needed

Design decisions the prototype made — confirm or change

Visual choices the storyboard quietly committed to. Each is reversible cheaply now and expensively later.

q · 01 — provenance chip placement

The "handoff · path b · opus 4.6 · 2,841 chars" chip lives just below the thread title header, above the first message. Alternative: as a hover-only popover on a small icon, so it doesn't add a permanent row.

prototype picked: always-visible header chip · alt: hover-only icon

q · 02 — banner placement and dismissibility

Fallback banner sits between the thread frame bar and the first message, dismissible via ×. Alternative: floats as a toast at top-right that auto-dismisses; less prominent but easier to miss.

prototype picked: inline dismissible banner · alt: toast notification

q · 03 — composer pre-fill rendering

Pre-filled user-msg fork text is rendered italic in the composer textarea to signal "this came from elsewhere." Alternative: render upright like normal typed text (no signal), or above the textarea as a quoted block with the textarea below.

prototype picked: italic in-textarea · alt: quoted block above textarea

q · 04 — internal turn visibility for power users

The "show internal" debug toggle (scenario 05 frame 02) exposes path A's hidden turns. Lives in the dev menu only. Alternative: hide entirely behind MCODE_DEBUG=1 env var; alternative: surface as a "view session state" command palette entry.

prototype picked: dev-menu toggle · alt: env-gated, alt: command palette

q · 05 — fork tooltip copy by anchor role

Hover tooltip says "retry from here" on user messages, "continue from here" on assistant messages. Surfaces the semantic difference but adds copy to memorize. Alternative: single "branch from here" label, derive intent from context.

prototype picked: role-aware copy · alt: uniform "branch"

q · 06 — child thread shows "forked from" trail

Thread title bar shows "forked from t_a4f2" as a clickable link back to the parent. Useful for navigation; takes space. Alternative: bury in thread settings.

prototype picked: clickable in title bar · alt: settings-only

q · 07 — slash command source of truth in fork composer

When the fork composer has its own provider override, the slash palette reads from composerProviderOverride ?? thread.provider. Alternative: always read from the composer's current provider selection even on non-fork composers (broader fix, more places to test).

prototype picked: fork-scoped override · alt: composer-wide refactor

q · 08 — fork rename scope

UI copy switches everywhere: "Branching from" → "Forking from", branch tooltip → "Fork from here", icon swaps from GitBranch to GitFork. Component names stay (ComposerBranchBar, branchFromMessageId prop) for minimal blast radius. Alternative: full rename including component identifiers and props — bigger refactor, cleaner long-term.

prototype picked: copy + icons only · alt: full code-level rename
